# Per-board bring-up test summary

Live record of the test-runner results, one block per physical board.
Generated by the `test_report` module in `src/test_report.{c,h}` —
the RTT log block at the end of each `west flash` run gets pasted
verbatim below, with a small header noting build date / hardware
state / any per-board operator notes.

For full bring-up history and the design context behind each
test, see [`steps.md`](./steps.md). For the canonical Kconfig +
dts setup that produces these results, see `prj.conf`,
`sysbuild.conf`, and the dtsi in `boards/palroc/nrf9151dk/`.

## Test-environment caveats

Worth knowing when reading these results — some of the "the board crashed
the host" observations from earlier sessions are *not* attributable to the
board:

- **The test laptop's USB-C ports were destroyed by a short several weeks
  ago.** Only the legacy USB-A controller is alive, and it now carries
  every J-Link / PPK2 / hub / peripheral. That one surviving controller is
  load-stressed and tips over more easily than a healthy laptop would.
  USB drops observed during board testing are likely a mix of board-side
  events AND the laptop's degraded USB capacity giving up — not always
  the board's fault.
- **Recommendation for continuing work**: powered USB hub (or USB
  isolator) between the laptop and any embedded device. Decouples the
  diagnostic — any post-hub USB drops are now provably the laptop, not
  the board.
- **PPK2 source mode is reproducibly unsafe on v1 boards** (see board
  3's notes below): triggers npm1300 BUCK2 = 4.2 V over-volt. Power
  v1 boards from battery only. PPK2 inline-measure mode + battery is the
  safe configuration if you need current data.

## Known validation gaps (as of 2026-05-05)

These items were **not validated** during bring-up and must be addressed by the application firmware engineer:

### 1 — 9151 test requires L15 to be erased

Running the 9151 test firmware only works reliably with **the L15 flash erased** (or L15 held in reset). The two chips share SPI3, i²c2, and inter-MCU signal pins; if L15 firmware is active and driving those pins, it causes bus contention that corrupts 9151 peripheral reads. Until a software bus-arbitration protocol is in place, always erase the L15 before testing the 9151 side and vice versa.

### 2 — Wi-Fi validated via 9151 only; 2.4 GHz APs not seen; RF switch power path not end-to-end

The nRF7000 scan was run from the 9151 with the RF switch VC **manually wired to VCC** on the bench. Only **5 GHz APs** were visible — 2.4 GHz APs were absent, likely due to the RF switch not being in the correct routing state without software control of SW_CTRL0.

The production RF path requires:
- **L15 P1.10 (RFFE_BLE_WIFI_ENABLE) HIGH** → RF switch VDD powered (L15 firmware now does this at boot).
- **SW_CTRL0** set to the Wi-Fi antenna position (which chip drives this and how is still an open question — see steps.md 16.9.2).

The full integrated path (L15 running + 9151 scanning + both bands visible) was never validated. A re-test with both firmwares running and SW_CTRL0 correctly controlled is required to confirm the 2.4 GHz path and the end-to-end RF switch sequencing.

### 3 — inter-MCU UART (LP-UART) not validated

The nrf-sw-lpuart link between the 9151 (UART0) and the L15 (uart30) was tested but **never passed**. The L15 side received the REQ/RDY handshake but zero data bytes — root cause is a deprecated `UART_0_NRF_HW_ASYNC` Kconfig approach in `prj.conf` that does not correctly arm the 9151-side async TX in NCS 3.3.0. See steps.md 16.9.3 for the exact fix (DT `timer = <&timer2>` phandle approach).

## Late-session findings and firmware updates (2026-05-05)

### SHPHLD + Hall sensor design clarification

Sensor: **DRV5032FBDBZR** (TI, omnipolar, push-pull, SOT-23-3). Bench-verified output: no magnet → HIGH, magnet present → LOW. The nPM1300 SHPHLD exits ship mode when pulled LOW — these are natively compatible: place magnet → DRV5032 output LOW → SHPHLD LOW → PMIC powers up. No logic inversion needed. DRV5032 must be powered from VSYS so it operates during ship mode (~1.8 µA, negligible shelf drain).

**Root cause of the Q4 reset loop (v1 schematic):** Q4 was wired as an inverter between the Hall sensor and SHPHLD, making the polarity backwards. **v2 fix: remove Q4, connect DRV5032 output directly to SHPHLD, power DRV5032 from VSYS.** See steps.md 16.7 for the full analysis.

### MPPT validated — wrong passive values root cause found and fixed

The nPM1300 MPPT charger input was previously unvalidated. Root cause: the MPPT threshold resistor divider values in the v1 schematic were wrong, setting the trigger voltage too low for MPPT to ever engage. After correcting the passive component values on the bench, MPPT triggers correctly and the charger operates as intended. **No firmware change required — hardware fix only.** v2 schematic must use the corrected resistor values.

### GNSS LNA — replacement sourced

The LNA was marked ⚠ in the step 16.5 validation table after the VBUS incident exposed it to 5 V (Vmax = 3.4 V). **Replacement LNA is now in hand** (2026-05-05). Re-solder and re-run `RUN_GNSS_PROBE=1` to confirm the GNSS path is back to the pre-incident baseline (12 SVs / 41 dB-Hz). The firmware and COEX0/LDO1 configuration are unchanged — only the physical component needs to be replaced.

### L15 firmware update — `0.3.0-l15-rffe-enable`

The L15 firmware was updated (commit `59c496f`) to drive **P1.10 (RFFE_BLE_WIFI_ENABLE) HIGH** at boot via a `SYS_INIT` at POST_KERNEL priority 50. This pin gates the BGS12SN6E6 RF switch VDD through an RV2C010UNT2L gate FET; without it, the RF switch has no power and the antenna is not routed.

The previous test results for boards 3 and 4 were collected with firmware `0.2.0-l15-testreport`, which did **not** drive P1.10. The RF switch VDD was supplied during those tests by the manual VC wire to VCC, not by L15 firmware. With `0.3.0-l15-rffe-enable` the firmware takes ownership of this signal correctly.

**This firmware has not yet been flashed and re-tested on hardware.** Re-flashing boards 3/4/5 with `0.3.0-l15-rffe-enable` and re-running the Wi-Fi scan (with both 9151 and L15 active, and SW_CTRL0 correctly set) is the next validation step for the integrated RF path.

### Board 3 — L15 sub-app (`nrf54l15/`)

- **Date**: 2026-04-30
- **Firmware**: `0.2.0-l15-testreport`
- **Power**: **battery only** (PPK2-as-source reproducibly triggers the step-14 BUCK2-overvolt fault — see below)

```
=== L15 BOARD 3  fw=0.2.0-l15-testreport  TEST SUMMARY ===
  [PASS] boot                  kernel init OK, K32SRC_RC fix held
  [PASS] gpio_park             P1.4(SDA)=1  P1.5(SCL)=1  (high-Z, levels passive)
  [PASS] ble_init              bt_enable OK
  [PASS] ble_advertise         non-connectable, name="palroc-l15"
=== L15 BOARD 3 totals: PASS=4  FAIL=0  INFO=0  SKIP=0 ===
```

**Notes**:
- L15 BLE path works end-to-end **only when battery-powered**.
- **PPK2-source-mode on board 3 reproducibly triggers the step-14 BUCK2-overvolt fault**: BUCK2 reads ~4.2 V (passing VSYS) instead of the configured 2.9 V. With BUCK2 at 4.2 V, every chip on that rail (including the L15 itself) is over-spec, and the L15 BLE either fails to start or is unstable.
- Battery-source vs PPK2-source is the only variable that changes; same firmware, same dts, same hardware. Strong evidence that something about how PPK2 ramps / sources VSYS interacts badly with the npm1300's regulation loop on this board (probably the floating MPPT input from step 14, possibly compounded by hall1's fault). Battery's slower ramp + lower-impedance source doesn't trigger it.
- **Operational rule on board 3 (and all v1 boards until proven otherwise): power from battery only**. PPK2 in source mode is a known-bad path on this hardware. Use PPK2 inline-measure mode + battery if you need current measurements.
- This finding belongs to step 14's hypothesis chain (hall1 + MPPT-floating + npm1300 regulation interaction). Doesn't change the v2 deltas (already include "tie VPV to ground" and "investigate hall1 footprint"); does add the operational warning **"never use PPK2 source mode on v1 boards"** to the bring-up checklist.
